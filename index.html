OOP Concepts in NOVƏ

This online shopping system is architected using Object-Oriented Programming principles — a demonstration of how real-world software is built using classes, inheritance, polymorphism, and encapsulation.

🧱

Classes & Objects

Foundation of OOP

The system uses classes like Product, Cart, Order, User, and Category. Each product on the page is an instance (object) of the Product class.

class Product {
  constructor(id, name, price, category) {
    this.id = id;
    this.name = name;
    this.price = price;
    this.category = category;
  }
  getDisplayPrice() {
    return `₹${this.price.toLocaleString()}`;
  }
}
🧬

Inheritance

IS-A Relationship

Electronics, Fashion, and HomeDecor all extend the base Product class. They inherit common attributes and add specialized behavior like warranty or size selection.

class Electronics extends Product {
  constructor(id, name, price, warranty) {
    super(id, name, price, 'Electronics');
    this.warranty = warranty;
  }
  getWarrantyInfo() {
    return `${this.warranty} Year Warranty`;
  }
}
class Fashion extends Product {
  constructor(id, name, price, sizes) {
    super(id, name, price, 'Fashion');
    this.sizes = sizes;
  }
}
🔒

Encapsulation

Data Hiding & Bundling

The Cart class encapsulates its internal item list. External code cannot directly mutate it — only through controlled methods like addItem(), removeItem(), and getTotal().

class Cart {
  constructor() {
    this.#items = []; // private
  }
  addItem(product, qty = 1) {
    const existing = this.#items.find(
      i => i.product.id === product.id);
    if (existing) existing.qty += qty;
    else this.#items.push({product, qty});
  }
  getTotal() {
    return this.#items.reduce(
      (sum, i) => sum + i.product.price * i.qty, 0);
  }
}
🔀

Polymorphism

One Interface, Many Forms

The renderCard() method is overridden in each subclass to display category-specific UI — Electronics shows warranty badge, Fashion shows size selector, etc.

// Base class method
class Product {
  renderCard() {
    return `<div>${this.name}</div>`;
  }
}

// Overridden in subclass
class Electronics extends Product {
  renderCard() { // override
    return `${super.renderCard()}
      <span>${this.warranty}yr warranty</span>`;
  }
}
🎭

Abstraction

Hiding Complexity

The PaymentProcessor abstract class defines the interface for all payment methods. Users just call pay(amount) — the concrete implementation varies by method (UPI, Card, COD).

// Abstract base
class PaymentProcessor {
  pay(amount) { /* abstract */ }
}

class UPIPayment extends PaymentProcessor {
  pay(amount) {
    processUPI(amount); // UPI logic
  }
}
class CardPayment extends PaymentProcessor {
  pay(amount) {
    processCard(amount); // Card logic
  }
}
🔗

Association & Aggregation

Object Relationships

An Order has-a User, a list of CartItems, and a PaymentProcessor. This models real-world relationships between system entities.

class Order {
  constructor(user, cart, payment) {
    this.user = user; // Association
    this.items = [...cart.getItems()]; // Aggregation
    this.payment = payment;
    this.orderId = Order.generateId();
    this.total = cart.getTotal();
    this.status = 'confirmed';
  }
}
